Why This Shape

Moran's lesson is that static cascades expose the control problem. The hollow nodes are places where the adviser would ask for a new pattern or scan, not places to hand-maintain a list.

Forward-Model Story: Held / Deferred Schema

The held/deferred schema node is the clearest place in this slice where the graph is informed mainly by expected patterns rather than by existing apparatus. The story is still concrete: if this schema existed, every parked obligation would become a typed object with a reason, owner, evidence condition, wake trigger, expiry or review rule, and re-entry path.

What We Would Get

  • Bounded deferral memory. Held work would stop living as folklore in mission prose or chat history; every pause would have a durable record and a named condition for being revisited.
  • Operator-safe re-entry. The operator could ask what should wake up now, and the answer would be governed by evidence, dates, or dependency events rather than by manual archaeology.
  • Mission pressure signal. Repeated held items with the same reason or missing evidence would become visible pressure for new missions, invariants, or substrate work.
  • Pattern-seeding loop. As deferrals accumulate, repeated reason/trigger/outcome shapes would seed real library patterns rather than remaining one-off exceptions.

Candidate Patterns Seeded By The Story

  • agent/held-item-wake-trigger: a held item is well formed only if it names the event, evidence, or date that wakes it back into work.
  • agent/deferral-ledger-as-operator-memory: deferred obligations should live in a queryable ledger rather than in prose-only mission margins.
  • mission/deferred-work-reentry-protocol: when a held item wakes, it must re-enter through a declared mission phase or operator review gate rather than becoming an unscoped task.
