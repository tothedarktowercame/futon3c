# E-typed-holes-aif-alternate — M-typed-holes down the HEAD→AIF lifeform path

**Paired with `E-typed-holes-example-self.md`. A counterfactual: what
M-typed-holes *would have looked like* had we seeded it as an AIF+ lifeform
(HEAD → sigil → vitals → … → ARGUE-as-aif⁺-battle) the way E-mission-head did,
instead of the IDENTIFY-first "as planned" route we actually took. The point is
the comparison (§5): does the AIF route reach the same design, and what does it
surface that the standard ARGUE missed? Method mirrored from
`futon6/holes/missions/E-mission-head.{md,aif.edn}` + `anatomy-of-a-futonic-mission.md`.**

## 1. The HEAD that wasn't (recast as a design pattern — AIF terminal vocab)

- **IF:** the stack keeps building the same object — *a slot waiting to be
  filled* — six times over (a query waiting for an answer, a `sorry` waiting for
  a proof, an ungrounded symbol waiting for a binder, a comb hole, a hungry mined
  node, a `type=query` bell), and "fill the slot" is one act done six ways.
- **HOWEVER:** they are built separately; the shared datatype (typed hole) and
  shared operation (fill) are invisible; there is no shared query layer; and ArSE
  — the Q&A substrate that *should* be the witness store — sits starved.
- **THEN:** name the **one** datatype (a typed hole = a Poly position with typed
  directions) and the **one** operation (fill), with the six surfaces as
  projections; record every fill in ArSE so it is witnessed.
- **BECAUSE:** one operator is deterministic and testable where six are racy and
  un-auditable; *asking a question and proving a theorem become the same act*;
  and a self-representing stack needs its central datatype to be a single thing it
  can point at itself.

## 3. Birth vitals + failure conditions (the AIF-unique artifact)

The organism reading produces a **health-at-birth** readout *and* — the part the
"as planned" route never wrote — **stated failure conditions / a death clause**:

- **Health (would-be):** `construction-state` strong (unlike E-mission-head's
  4/3 open, M-typed-holes is born late — the datatype is already 9/10 formalised
  in DarkTower); `bit-confidence [compute]`; reading: *"unusually healthy at
  birth — the actionable half is already built; the risk is not feasibility but
  adoption."*
- **Failure conditions (the up-front discipline):**
  1. **dead if** no projection actually routes through the single `fill` (it stays
     six implementations forever — the unification is decorative);
  2. **dead if** ArSE stays starved — fills are not witnessed, so "proof =
     witnessed fill" never lands;
  3. **dead if** the query layer never runs over a real store (substrate-2a / MO)
     — queries-as-scopes stays a demo.
- **Death clause:** declare M-typed-holes dead, publicly, if none of the six
  projections is re-routed through the shared `fill` **and** no fill is witnessed
  in ArSE by an agreed deadline.

## 4. ARGUE as an aif⁺ battle (the mission argues its own existence)

Per E-mission-head's contest substrate (`spine`/`nodes`, `run_aif2_contest.clj`):
the mission defends its thesis against the strongest objections we can write.

- **Thesis (n0):** one typed-hole datatype + one witnessed `fill`, six
  projections, makes asking = proving.
- **Objection A — "just renaming; six fills work."** Rebuttal: the six already
  *disagree* (grounding has precision gates, the comb didn't, the query's
  `answers↔fill` was definitional) — disunity caught by audit, not prevented.
  `single-authority-registration` shows pre-hoc one-authority beats post-hoc
  reconciliation. **Survives.**
- **Objection B — "premature abstraction."** Rebuttal: not premature — the
  datatype is already *forced* (it's `PFunctor`, free from mathlib) and already
  *populated* (satiety). Abstracting a thing the data already exhibits is
  description, not speculation. **Survives.**
- **Objection C — "ArSE is already starved; another typed thing won't help."**
  Rebuttal — and this is the sharp one: this objection is **failure-condition #2
  turned into the design driver**. The mission only lives if it *feeds* ArSE
  (witnessed fills), so adoption is not a footnote but the death clause.
  **Survives only by committing to the witness loop** — which reweights the
  design (see §5).

## 5. The comparison (the payoff)

| | "as planned" route (done) | AIF-driven route (this doc) |
|---|---|---|
| entry | IDENTIFY-first (the gap + exemplars) | HEAD-first (intent compiled to a sigil) |
| what justifies the design | pattern cross-ref + technical coherence | the organism's failure conditions / death clause |
| adoption risk | one decision among six (D4/D5) | **the primary driver** (failure cond. #2) |
| health visible at birth? | no (known only retrospectively) | yes (sigil + vitals at seeding) |
| verdict on the design | *inevitable* | **same design — reached independently** |

Three findings:

1. **Both routes converge on the same design** (one `fill`, witnessed, six
   projections). That's a strong result: the design is robust to the derivation
   path — the "inevitable" claim is now backed by *two independent arguments*,
   not one.
2. **The AIF route surfaces what "as planned" missed: the failure conditions /
   death clause** — and they are not new risks, they are the MAP "missing" list
   and the M-typed-bells "ArSE starved" gap, *promoted to up-front survival
   conditions*. The as-planned ARGUE treated witnessing as decision D4; the AIF
   route makes it the thing the mission *lives or dies by*. Same fact, different
   weight — and the AIF weight is more honest about what actually matters
   (adoption, not feasibility).
3. **The AIF route would have written a HEAD** — the node the self-typing
   (`E-typed-holes-example-self.md`) flagged as the missing meta-hole. So the two
   excursions agree: M-typed-holes' real gap is the absent HEAD + failure
   conditions, and the AIF route is precisely the one that fills it.

## 6. Verdict — carry back to the real mission

Adopt from the AIF route into M-typed-holes proper (Joe's deferred plan): **(1)**
write the HEAD (§1 above is the draft); **(2)** add the failure conditions +
death clause (§3) to the charter — they are the missing adoption gate; **(3)** do
*not* re-run DERIVE — both routes reached the same design, so re-derivation would
only confirm it (record that as the finding, not redo the work). The genuinely
new artifact the AIF route adds is the **death clause**, which the as-planned
route had no place for.
